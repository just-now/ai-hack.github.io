SIESTA

Self-Iterating Engineer for Stress-Test Anomalies

Make the system debug itself.

The problem

Stress-testing is cheap.
Diagnosing is archaeology.

A stress test injects faults — partitions, crashes, slow disks, whatever the harness supports.

Each run emits tens of thousands of interleaved log lines.

Livelock. Deadlocks. Perf cliffs. Silent stalls. They hide in the noise.

engineer reads logs → hypothesizes → patches → repeat

The bottleneck isn't the bug rate. It's human attention.

The idea

What if the test loop closed itself?

Three LLMs in the loop.
One reads. One writes. One gates.

Don't ask what happened.
Show the evidence.

Cycle: ~3 minutes. Run it overnight.
Wake up to a branch full of forensics.

The loop

One orchestrator, two specialists

The agents

Read. Write. Verify.

Therapist

The diagnostician.

  • Reads the test logs directly
  • Ranks hypotheses with evidence chains
  • Open-ended methodology, not a checklist

Not vibes — chains.

Surgeon

The engineer who acts.

  • Reads the report, edits minimally
  • Runs the build & tests
  • Full report verbatim in the commit body

git log -p = forensic archive.

Validator

The independent oracle.

  • Looks at the rerun, not the patch
  • Returns OK — fix sticks
  • Returns ERROR(context) — feeds next loop

Never heals. Only judges.

One reads. One writes. One gates. Together, a loop that self-corrects.

Design

Choices that pay rent

↻
Single long-running session One coherent transcript. No per-tick spin-up.
≡
Inline sub-agent prompts One file (orchestrator.md) is the whole system.
⇄
Synchronous handoff Therapist → Surgeon → Validator. Never two ticks racing.
▦
Filesystem as state machine Overnight runs stay bounded in context.
✎
Reports embedded in commits git blame answers why, not just what.
⊕
History is additive Coder never rewrites. Every iteration is reviewable.
Out of scope

What we're not building

Discipline by subtraction.

✕ Auto-merge

Agents draft fixes. Humans still review and merge.

✕ New tooling

The whole loop = markdown + shell + two Task calls.

The smallest viable thing that could find a real bug.

Open questions

What we'll learn

  • Hit rate. Real anomalies per hour of test time?
  • False positives. How often does the analyzer cry wolf?
  • Mergeability. Are the coder's diffs reviewable, or just plausible?
  • Trace utility. Does report-in-commit speed up the next human investigation?
  • Long tail. What surfaces overnight that we'd never catch by hand?

None of these have answers from the design. Only running it does.
